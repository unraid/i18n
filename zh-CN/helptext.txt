#### 分配设备
unRAID 磁盘阵列由若干数据磁盘和最多两个奇偶校验磁盘组成。

数据
磁盘专门用于存储用户数据，而奇偶校验磁盘提供必要的冗余以从任何单盘或双盘故障中恢复。

请注意，在提到阵列存储设备时我们仔细使用术语*磁盘*。

我们

在提到实际的硬盘（HDD）设备时使用 *硬盘*（或有时仅使用 *驱动器*）这一术语。这是因为在 RAID 系统中，可能会读取/写入相应硬盘已禁用甚至丢失的阵列磁盘！

此外，可以询问"哪个设备被分配为奇偶校验磁盘？"；或者，"哪个设备对应于 disk2？"。

因此，我们需要一种方法将硬盘分配给阵列磁盘。

在阵列停止时可以在主页面中完成此操作。
每个阵列磁盘都有一个下拉框列出所有未分配的设备。

要分配设备，只需从列表中选择即可。 每次进行设备分配时，系统都会更新配置文件以记录分配情况。
#### 要求
与传统 RAID 系统不同，后者将数据条带化在所有阵列设备上，Unraid 服务器将文件存储在各个硬盘中。

因此，所有文件写入操作都将涉及文件正在写入的数据磁盘和奇偶校验磁盘。  因此，
* 奇偶校验磁盘的大小必须与任何数据磁盘一样大或更大，并且
* 在有选择的情况下，奇偶校验磁盘应是您集合中最快的磁盘。 #### 指导方针
以下是您在设计 unRAID 磁盘阵列时应遵循的步骤： 1.
决定将哪个硬盘用作奇偶校验，以及您将用作数据磁盘1、磁盘2等的硬盘，并以某种方式标记它们。

此外，请查找每个硬盘的序列号，并将其记下；您稍后将需要此信息。 2.
安装您的硬盘设备，启动服务器并打开 webGUI。 如果这是新系统构建，则主页将显示未安装磁盘。
这并不意味着系统无法检测到您的硬盘；这只是意味着尚未分配任何硬盘。 3. 记得您在步骤1中记录的序列号吗？对于奇偶校验和每个数据磁盘，根据序列号从下拉列表中选择合适的硬盘。
#### 热插拔

如果您的硬件支持，您还可以将硬盘*热插拔*到服务器中。

例如，
如果您使用硬盘笼，可以在服务器通电且阵列停止时将它们插入服务器。 刷新主页以将新的未分配设备显示在分配下拉列表中。

强烈建议仅使用前128个字符中的95个可打印字符[ASCII表](https://en.wikipedia.org/wiki/ASCII)，因为它们的二进制表示始终相同。 其他字符的编码可能因系统配置而异，更换编码后，您的口令将无效。
如果希望使用更长的口令或包含二进制数据，请上传密钥文件。

有关*安全*口令的标准，请参阅 [cryptsetup FAQ](https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects)。 **请牢记**此口令。 **如果丢失，加密内容将无法恢复！**

如果您使用的是 Microsoft 帐户，最好只在 Unraid 中创建一个用户名简单的用户并设置密码，然后在 Windows 中转到控制面板 -> 凭据管理器 -> Windows 凭据 -> 添加 Windows 凭证并输入正确的 Unraid 服务器名称和凭证。

`server:`<br>
`private-domain: "myunraid.net"`

**Ubiquiti USG 路由器：**您可以添加此配置行：

`set service dns forwarding options rebind-domain-ok=/myunraid.net/`

**OpenDNS:** 转到“设置”->“安全性”并 *取消选中* 
  "可疑响应 - 阻止内部 IP 地址"。 这是一个全有全无的设置。

:nfs_tunable_fuse_remember_help:

启用 NFS 时，您可以使用此可调参数缓解或解决使用 NFS 客户端时遇到的“NFS 陈旧文件句柄”问题。
实质上，(fuse_remember) 告诉内部子系统（称为 "fuse"）需要多长时间"记住"或"缓存"与用户共享相关的文件和目录信息。当 NFS 客户端尝试访问 服务器上的文件（或目录）时，并且该文件（或目录）名未缓存时，您可能会遇到
"陈旧文件句柄"的问题。

此可调参数的数值为缓存文件/目录名称条目的秒数，默认值为 330 表示 5 分半钟。您还可以设置两个特殊值
这个计算结果是：
* 0 表示不缓存文件/目录名称，
* -1 表示永久缓存文件/目录名（或直到阵列停止）

如果您启用了 NFS，但仅计划将其用于磁盘共享，而不是用于用户共享，则将其设置为 0 会很合适。
如果没有其他超时似乎可以解决客户端上的“陈旧文件句柄”，则将其值设置为 -1 会比较合适。
请注意，设置 -1 的值将使内存占用量按每个缓存的文件/目录名增长约 108 字节。取决于您的服务器上安装了多少 RAM 以及通过 NFS 访问了多少

文件/目录，这可能会也可能不会导致内存不足的情况。
:end

:eth_enable_bonding_help:
绑定是一种将多个物理以太网接口组合成单个*绑定*接口**bond0**的功能。
这可以用来提高系统的连接冗余性和/或吞吐量。

:eth_bonding_mode_help:
**模式 0（balance-rr）**<br>
这种模式从第一个可用的从接口依次传输数据包。
如果两个真实接口是绑定中的从接口，而两个数据包到达时，定向出绑定接口，第一个将在第一个从接口上传输，第二个帧将在第二个从接口上传输。
第三个数据包将发送到第一个，依此类推。 这提供了负载平衡和容错。

**模式 1（主动-备份）- 默认**<br>
这种模式将其中一个接口置于备份状态，只有当主动接口丢失链接时才会激活。
在一次时间内，只有一个从接口是活动的。 仅当活动从接口失败时，不同的从接口才会变为活动。
这种模式提供容错。

**模式 2（balance-xor）**<br>
这种模式基于 XOR 算式来传输数据包。 源 MAC 地址与目标 MAC 地址模从接口数量 XOR。
这为每个目标 MAC 地址选择相同的从接口，提供负载平衡和容错。

**模式 3（广播）**<br>
这个模式在所有从接口上传输所有内容。 这种模式最少使用（仅用于特定目的），仅提供容错。

**模式 4（802.3ad）**<br>
这种模式被称为*动态链接聚合*。 它创建共享相同速度和双工设置的聚合组。
它需要支持 IEEE 802.3ad 动态连接的开关。
传输流量的从接口选择是根据传输哈希策略进行的，可以通过 xmit_hash_policy 选项更改默认简单 XOR 策略。
请注意，并非所有传输策略都可能符合 802.3ad，特别是在 802.3ad 标准第 43.2.4 节的包误排序要求方面。
不同的对等实施可能对不合规有不同的宽容度。

**模式 5（balance-tlb）**<br>
这种模式称为*自适应传输负载平衡*。 输出流量按照每个从接口上的当前负载和队列分布。
输入流量由当前的从接口收到。

**模式 6（balance-alb）**<br>
这种模式称为*自适应负载平衡*。 这包括针对 IPV4 流量的 balance-tlb + 接收负载平衡（rlb）。
接收负载平衡通过 ARP 协议实现。
绑定驱动程序拦截服务器发送的 ARP 回复，将源硬件地址覆盖为绑定中从接口的唯一硬件地址 
这样的设置使得不同的客户端会使用绑定中不同从接口的硬件地址。

*模式 1（主动-备份）是推荐的设置 其他模式允许您设置特定环境，但可能需要适当的交换机支持。
选择不支持的模式可能导致通信中断。
:end

**注意：** bash 历史记录将被写入 USB 启动设备，因此这会导致更高的磨损。

**注意：** 禁用和启用将移除整个 bash 历史记录。
:end

:WOL_intro_help:
此页面允许设置以使用WOL魔法包启动/恢复和停止虚拟机、容器（Docker和LXC）

它不设置Unraid服务器的唤醒。

该过程将查找服务（虚拟机或容器）中定义的MAC地址，除了dockers之外。 Docker 在运行之前没有 MAC 地址，因此您需要定义用户 MAC 地址以允许您启动它们。

如果服务暂停，WOL包将恢复。 

对于每个服务，您可以设置：启用、禁用或启用并关闭。

当启用并关闭设置时，如果该服务正在运行，当接收到下一个WOL数据包时，将执行关机。
:结束

:WOL_run_VM_help:

:结束

:WOL_run_LXC_help:
如果设置为是，当接收到唤醒局域网数据包时，将对 LXC 进行检查，否则将忽略 LXC。 需要安装 LXC 插件以处理 LXC。
:结束

:WOL_run_shutdown_help:
如果设置为是，则接收到局域网唤醒数据包时将进行检查，如果启用了服务并且该服务正在运行，则实体将关闭。 系统必须设置为启用和关闭才能操作。
