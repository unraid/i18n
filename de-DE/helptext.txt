**Pool-Geräte** ist ein einzelnes Gerät oder ein Pool mehrerer Geräte *außerhalb* des unRAID-Arrays.  Es kann für den Netzzugriff exportiert werden, genau
wie ein Array-Gerät.  Da es sich außerhalb des unRAID-Arrays befindet, ergibt sich ein erheblich schnellerer Schreibzugriff.

Es gibt zwei Möglichkeiten, die Pool-Geräte zu konfigurieren:

1. Als einzelnes Gerät oder
2. Als Multi-Geräte-Pool.

Wenn als einzelnes Gerät konfiguriert, können Sie das Gerät in jedem unterstützten Dateisystem formatieren (btrfs, reiserfs,
oder xfs).  Diese Konfiguration bietet die höchste Leistung, jedoch auf Kosten des fehlenden Datenschutzes - wenn das
einzelne Poolgerät ausfällt, können alle darauf enthaltenen Daten verlorengehen.

Wenn ein Multi-Geräte-Pool konfiguriert ist, wählt das Unraid-OS automatisch das Format *btrfs-raid1* (für Daten
und Metadaten).  btrfs erlaubt, beliebig viele Geräte zum Pool hinzuzufügen, und jede Datenkopie wird garantiert
auf zwei unterschiedliche Geräte geschrieben.  Daher kann der Pool ein einzelnes Festplattenversagen ohne Datenverlust überstehen.

Wenn [Benutzerfreigaben](/Settings/ShareSettings) aktiviert sind, können Benutzerfreigaben konfiguriert werden, um
automatisch das Pool-Gerät zu verwenden, um
Schreibzugriffe zu beschleunigen.  Ein spezieller Hintergrundprozess namens *Mover* kann geplant werden, um zu laufen
um Benutzerfreigabedateien periodisch von dem Cache auf das Array zu verschieben.

#### Geräte zuweisen

Ein unRAID-Disk-Array besteht aus einer Anzahl von Datenträgern und bis zu zwei Paritätsdatenträgern. Die Daten Datenträger werden ausschließlich zur Speicherung von Benutzerdaten verwendet, und der/die Paritätsdatenträger bietet/bieten die notwendige Redundanz
um jede einzelne oder doppelte Festplattenstörung wiederherzustellen.
Beachten Sie, dass wir sorgfältig den Begriff *Datenträger* verwenden, wenn wir auf ein Array-Speichergerät verweisen.

Wir  verwenden den Begriff *Festplatte* (oder manchmal einfach *Laufwerk*), wenn wir auf eine tatsächliche Festplatte (HDD) Bezug nehmen.
Gerät.
Dies liegt daran, dass es in einem RAID-System möglich ist, auf einen Array-Datenträger zu lesen/schreiben, dessen entsprechende Festplatte deaktiviert oder sogar fehlend ist!
Zusätzlich ist es praktisch zu wissen, "welches Gerät ist dem Paritätsdatenträger zugewiesen?" oder "welches Gerät entspricht disk2?". Wir benötigen daher eine Möglichkeit, Festplatten auf Array-Datenträger zuzuweisen.
Dies wird hier auf der Hauptseite erreicht, wenn das Array gestoppt ist.

Es gibt ein Dropdown-Feld für jeden Array-Datenträger, das alle aufgelisteten nicht zugewiesenen Geräte enthält. Um ein Gerät zuzuweisen, wählen Sie es einfach aus der Liste aus.
Jedes Mal, wenn eine Gerätezuweisung vorgenommen wird, aktualisiert das System eine Konfigurationsdatei, um die Zuordnung aufzuzeichnen. #### Anforderungen
Im Gegensatz zu herkömmlichen RAID-Systemen, die Daten über alle Array-Geräte verteilen, speichert ein Unraid-Server Dateien auf einzelnen Festplatten. Folglich beinhalten alle Dateischreibvorgänge sowohl den Datenträger, auf den die Datei geschrieben wird, als auch den/die Paritätsdatenträger.
Aus diesen Gründen,

* die Größe eines Paritätsdatenträgers muss so groß oder größer als einer der Datenträger sein,

und
* wenn möglich, sollte der/die Paritätsdatenträger das/die schnellste(n) Laufwerk(e) in Ihrer Sammlung sein. #### Richtlinien
Hier sind die Schritte, die Sie befolgen sollten, wenn Sie Ihr unRAID-Disk-Array entwerfen:  1.

Entscheiden Sie, welche Festplatte Sie für die Parität verwenden werden, und welche Festplatten Sie für

Datenträger 1, 2 usw. verwenden werden, und beschriften Sie sie auf irgendeine Weise.

Finden Sie außerdem die Seriennummer jeder Festplatte und notieren Sie sie sich, diese Informationen werden später benötigt.

2.

Installieren Sie Ihre Festplattengeräte, starten Sie Ihren Server und rufen Sie die webGUI auf.

Wenn dies ein neuer Systemaufbau ist zeigt die Hauptseite keine installierten Festplatten an.
Das bedeutet nicht, dass das System Ihre Festplatten nicht erkennen kann; es bedeutet nur, dass noch keine zugewiesen wurden.
3.

Erinnern Sie sich an die Seriennummern, die Sie in Schritt 1 notiert haben? Für Parität und jeden Datenträger wählen Sie die richtige Festplatte basierend auf ihrer Seriennummer aus der Dropdown-Liste aus.
#### Hot Plug Sie können auch *Hot-Plugging* von Festplatten in Ihren Server durchführen, wenn Ihre Hardware dies unterstützt.
Beispielsweise,

wenn Sie Festplattenkäfige verwenden, können Sie diese einfach in Ihren Server einstecken, während er eingeschaltet ist und das Array gestoppt ist. Aktualisieren Sie die Hauptseite, damit neue nicht zugewiesene Geräte in den Zuweisungs-
Dropdown-Listen erscheinen.

#### Nächste Schritte

Sobald Sie alle Ihre Festplatten zugewiesen haben, gehen Sie zu dem Abschnitt Array-Status unten und starten Sie das Array.

:info_btrfs_balance_help:

**Balance** führt das Programm *btrfs balance* aus, um die Extents über alle Pool-Geräte zu restrukturieren, zum Beispiel,
um den Pool von raid1 auf raid0 oder umgekehrt zu konvertieren.

Bei einer *vollen Balance* wird im Grunde genommen alles im aktuellen Dateisystem neu geschrieben.
Eine *Moduskonvertierung* betrifft die btrfs-Datenextents; Metadaten verwenden immer raid1 und werden bei Bedarf durch jede Balance-Operation zu raid1 konvertiert. Die Laufzeit kann je nach Dateisystemgröße und Geschwindigkeit des Geräts sehr lang sein.

Unraid-OS verwendet diese Standardoptionen beim Erstellen eines Multiple-Device-Pools:
`-dconvert=raid1 -mconvert=raid1`

Für vollständige Dokumentation siehe btrfs-balance [Manpage](https://man7.org/linux/man-pages/man8/btrfs-balance.8.html)
*Hinweis: raid5 und raid6 werden von der Linux-Community allgemein noch als **experimentell** angesehen*

:mgmt_use_ssl_tls_help:
Bestimmt, wie das webGUI auf HTTP- und/oder HTTPS-Protokolle in Ihrem LAN reagiert.

Wählen Sie **Nein**, um HTTP zu verwenden. Um auf Ihren Server zuzugreifen, verwenden Sie diese URL:

`http://<server-name>.<localTLD>`

oder diese URL:

`http://<server-ip-address>`

Wählen Sie **Ja**, um die Verwendung eines automatisch generierten selbstsignierten
SSL-Zertifikats zu aktivieren.  Verwenden Sie diese URL, um auf Ihren Server zuzugreifen:

`https://<server-name>.<localTLD>`

Beachten Sie, dass die Verwendung eines selbstsignierten SSL-Zertifikats eine Warnung im Browser generiert.
warnung.

Wählen Sie **Streng**, um die *ausschließliche* Verwendung eines myunraid.net SSL-Zertifikats
für https-Zugriff (siehe **Provision** unten) zu aktivieren.  Beachten Sie, dass ein DNS
Server erreichbar sein muss.

**Umleitungen:** Beim Zugriff auf `http://<server-ip-address>` oder `http://<server-name>.<localTLD>`, wird das 
Verhalten sich je nach Wert der SSL/TLS-Einstellung ändern.

* Wenn SSL/TLS auf **Streng** eingestellt ist, werden Sie zu `https://<lan-ip>.<hash>.myunraid.net` umgeleitet
* Wenn SSL/TLS auf **Ja** eingestellt ist, werden Sie zu `https://<server-ip-address> oder https://<server-name>.<localTLD>` umgeleitet
* Wenn SSL/TLS auf **Nein** eingestellt ist, wird die http-URL direkt geladen.

Wichtig: **Streng** kann möglicherweise nicht auswählbar sein, wenn Ihr Router oder der DNS-Server einen
[DNS-Rebinding-Schutz](https://en.wikipedia.org/wiki/DNS_rebinding) aktiviert hat. DNS-Rebinding
Schutz verhindert, dass DNS auf einen privaten IP-Netzwerkbereich aufgelöst wird. DNS-Rebinding-Schutz ist als
eine Sicherheitsfunktion in einem LAN gedacht, das möglicherweise veraltete Geräte mit fehlerhaften/unsicheren "Web"-Schnittstellen enthält.

Eine Quelle des DNS-Rebinding-Schutzes könnte Ihr ISP-DNS-Server sein. In diesem Fall kann das Problem behoben werden, indem
zu einem anderen DNS-Server wie OpenDNS gewechselt wird, bei dem DNS-Rebinding-Schutz ausgeschaltet werden kann.

Häufiger könnte der DNS-Rebinding-Schutz in Ihrem Router aktiviert sein. Die meisten Konsumentenrouter implementieren keinen DNS
Rebinding-Schutz, aber wenn sie dies tun, sollte eine Konfigurationseinstellung verfügbar sein, um es auszuschalten.

Höhere End- Router aktivieren normalerweise den DNS-Rebinding-Schutz. Typischerweise gibt es Möglichkeiten, ihn vollständig oder selektiv basierend auf der Domain auszuschalten.
Beispiele: **DD-WRT:** Wenn Sie "dnsmasq" mit aktiviertem DNS-Rebinding-Schutz verwenden, können Sie diese Zeile Ihrer Router

Konfigurationsdatei hinzufügen:
`rebind-domain-ok=/myunraid.net/`

**pfSense:** Wenn Sie den internen DNS-Resolver-Dienst von pfSense verwenden, können Sie diesen benutzerdefinierten Optionszeilen hinzufügen:

`server:`<br>

`private-domain: "myunraid.net"`
**Ubiquiti USG router:** Sie können diese Konfigurationszeile hinzufügen:

`set service dns forwarding options rebind-domain-ok=/myunraid.net/`

**OpenDNS:** Gehen Sie zu Einstellungen -> Sicherheit und *entfernen* Sie das Häkchen neben

"Verdächtige Antworten - Interne IP-Adressen blockieren".
 Es ist eine Alles-oder-Nichts-Einstellung. Wenn alles andere fehlschlägt, können Sie möglicherweise einen Eintrag in der *hosts*-Datei Ihres PCs erstellen, um externes DNS zu überschreiben und

direkt den FQDN Ihres myunraid.net-Servers zur lokalen IP-Adresse aufzulösen.
:end

:nfs_tunable_fuse_remember_help:

Wenn NFS aktiviert ist, kann dieser Eintrag verwendet werden, um Vorkommen von "NFS Stale File Handles" zu lindern oder zu lösen, die Sie mit Ihrem NFS-Client möglicherweise haben.
Im Wesentlichen teilt (fuse_remember) einem internen Subsystem (namens "fuse") mit, wie lange Dateien und Verzeichnisinformationen, die mit Benutzerfreigaben verbunden sind, "erinnert" oder "zwischengespeichert" werden sollen.  Wenn ein NFS-Client versucht, auf eine Datei (oder ein Verzeichnis) auf dem Server zuzugreifen, und der Dateiname (oder das Verzeichnis) nicht zwischengespeichert ist, könnten Sie auf "stale file handle" stoßen.
Der numerische Wert dieser Einstellbaren ist die Anzahl von Sekunden, Dateien/Verzeichnis-Nameneinträge zu zwischenspeichern,
wobei der Standardwert von 330 5 1/2 Minuten bedeutet.

Es gibt zwei Sonderwerte, die Sie auch festlegen können:
* 0, was bedeutet, dass Dateinamen/Verzeichniseinträge überhaupt nicht zwischengespeichert werden, und  * -1, was bedeutet, dass Dateinamen/Verzeichniseinträge für immer (oder bis das Array gestoppt wird) zwischengespeichert werden.
Ein Wert von 0 wäre angemessen, wenn Sie NFS aktivieren, es aber nur für Plattenfreigaben verwenden möchten, nicht Benutzerfreigaben.
Ein Wert von -1 wäre angemessen, wenn kein anderer Timeout das "stale file handle" auf Ihrem Client zu lösen scheint.

Seien Sie sich bewusst, dass das Festlegen eines Wertes von -1 dazu führt, dass der Speicherbedarf um ca. 108 Bytes pro zwischengespeicherter Dateinamen/Verzeichnisnamen zunimmt.
Je nachdem, wie viel RAM in Ihrem Server installiert ist und wie viele Dateien/Verzeichnisse Sie über NFS zugreifen, kann dies zu einem Arbeitspeichermangel führen oder auch nicht. :end

:eth_enable_bonding_help:
Bonding ist eine Funktion, die mehrere physische Ethernet-Schnittstellen zu einer einzigen *gebundenen* Schnittstelle namens **bond0** kombiniert.
Dies kann verwendet werden, um die Verbindungsredundanz und/oder den Durchsatz des Systems zu verbessern.
Es werden verschiedene Bonding-Modi unterstützt (siehe unten), aber einige Modi erfordern ordnungsgemäße Switch-Unterstützung.
:end

:eth_bonding_mode_help:
**Modus 0 (balance-rr)**<br>
Dieser Modus sendet Pakete in der Reihenfolge vom ersten verfügbaren Slave bis zum letzten.
Wenn zwei reale Schnittstellen Slaves im Bond sind und zwei Pakete ankommen, die an die gebundene Schnittstelle gesendet werden sollen, wird das erste am ersten Slave gesendet, und der zweite Frame wird am zweiten Slave gesendet.
Das dritte Paket wird am ersten und so weiter gesendet. Dies bietet Last-Balancing und Fehlertoleranz.

**Modus 1 (active-backup) - Standard**<br>
Dieser Modus setzt eine der Schnittstellen in einen Backup-Zustand und wird sie nur aktivieren, wenn der Link der aktiven Schnittstelle verloren geht.
Im Bond ist nur ein Slave zu einem Zeitpunkt aktiv. Ein anderer Slave wird nur aktiv, wenn der aktive Slave ausfällt.
Dieser Modus bietet Fehlertoleranz.

**Modus 2 (balance-xor)**<br>
Dieser Modus sendet Pakete basierend auf einer XOR-Formel. Die Quell-MAC-Adresse wird mit der Ziel-MAC-Adresse xoriert, geteilt durch die Slave-Anzahl.
Dies wählt den gleichen Slave für jede Ziel-MAC-Adresse aus und bietet Last-Balancing und Fehlertoleranz.

**Modus 3 (Broadcast)**<br>
Dieser Modus sendet alles auf allen Slave-Schnittstellen. Dieser Modus wird am wenigsten verwendet (nur für spezielle Zwecke) und bietet nur Fehlertoleranz.

**Modus 4 (802.3ad)**<br>
Dieser Modus ist als *Dynamic Link Aggregation* bekannt. Er erstellt Aggregationsgruppen, die die gleichen Geschwindigkeits- und Duplexeinstellungen teilen.
Es erfordert einen Switch, der IEEE 802.3ad dynamischen Link unterstützt.
Die Slave-Auswahl für ausgehenden Verkehr erfolgt gemäß der Hash-Politik für den Versand, die über die xmit_hash_policy-Option von der standardmäßigen einfachen XOR-Politik geändert werden kann.
Beachten Sie, dass nicht alle Übertragungsrichtlinien den Anforderungen des Abschnitts 43.2.4 des 802.3ad-Standards entsprechen, insbesondere in Bezug auf die Anforderungen zur Vermeidung von Paketfehlbestellungen.
Unterschiedliche Peer-Implementierungen haben unterschiedliche Toleranzen für die Nichteinhaltung.

**Modus 5 (balance-tlb)**<br>
Dieser Modus wird *Adaptive Transmit Load Balancing* genannt. Der ausgehende Verkehr wird entsprechend der aktuellen Last und Warteschlange auf jeder Slave-Schnittstelle verteilt.
Der eingehende Verkehr wird durch den aktuellen Slave empfangen.

**Modus 6 (balance-alb)**<br>
Dieser Modus wird *Adaptive Load Balancing* genannt. Dies umfasst balance-tlb + Empfangslastverteilung (rlb) für IPV4-Verkehr.
Die Empfangslastverteilung wird durch ARP-Aushandlung erreicht.
Der Bonding-Treiber fängt die ARP-Antworten, die vom Server gesendet werden, auf ihrem Weg ab und überschreibt die Quell-Hardwareadresse mit der eindeutigen Hardwareadresse eines der Slaves im Bond
sodass verschiedene Clients unterschiedliche Hardwareadressen für den Server verwenden.

*Mode 1 (Active-Backup) ist die empfohlene Einstellung. Andere Modi ermöglichen die Einrichtung einer spezifischen Umgebung, erfordern jedoch möglicherweise eine ordnungsgemäße Unterstützung des Switches.
Die Wahl eines nicht unterstützten Modus kann zu unterbrochener Kommunikation führen.
:ende

+ **APCsmart** - neuere serielle Zeichengeräte, geeignet für SmartUPS-Modelle, die ein serielles Kabel verwenden (nicht USB)
+ **Net** - Netzwerkverbindung zu einem Master-apcupsd über den Netzwerk-Informations-Server von apcupsd.
Dies wird verwendet, wenn die USV, die Ihren Computer überwacht, an einen anderen Computer angeschlossen ist. + **SNMP** - SNMP-Netzwerkverbindung zu einem SNMP-fähigen USV-Gerät
+ **Dumb** - altes serielles Zeichengerät zur Verwendung mit einfachen USVen
+ **PCnet** - PowerChute-Netzwerkabschaltungsprotokoll, das als Alternative zu SNMP mit der AP9617-Familie von Smart-Slot-Karten verwendet werden kann
+ **ModBus** - serielles Gerät zur Verwendung mit den neuesten SmartUPS-Modellen, die das MODBUS-Protokoll unterstützen
:ende

+ **apcsmart** - /dev/tty**

+ **Net** - hostname:port.
Hostname ist die IP-Adresse des NIS-Servers. Der Standardport ist 3551 + **Snmp** - hostname:port:vendor:community.
Hostname ist die IP-Adresse oder der Hostname der USV im Netzwerk. Vendor kann "APC" oder "APC_NOTRAP" sein. "APC_NOTRAP" deaktiviert das Abfangen von SNMP-Traps; normalerweise möchten Sie "APC". Port ist normalerweise 161. Community ist normalerweise "private" + **Dumb** - /dev/tty**
+ **PCnet** - ipaddr:benutzername:passwort:port.
ipaddr ist die IP-Adresse der USV-Verwaltungskarte. Benutzername und Passwort sind die Anmeldeinformationen, für die die Karte konfiguriert wurde. Der Port ist die Portnummer, auf der Nachrichten von der USV empfangen werden, normalerweise 3052. Es richtet nicht das Aufwecken des Unraid-Servers ein.

Der Prozess wird nach der definierten MAC-Adresse innerhalb des Dienstes (VM oder Container) suchen, mit Ausnahme von Dockern. Docker haben keine MAC-Adresse, bis sie ausgeführt werden, daher müssen Sie eine Benutzer-MAC-Adresse definieren, um sie zu starten. Wenn der Dienst pausiert ist, wird ein WOL-Paket fortgesetzt.

Für jeden Dienst können Sie einstellen: aktivieren, deaktivieren oder aktivieren und Herunterfahren. 

Wenn die Einstellung "aktivieren und Herunterfahren" gesetzt ist, wird dann, wenn der Dienst beim Erhalt des nächsten WOL-Pakets ausgeführt wird, ein Herunterfahren durchgeführt.

Wenn die Einstellung "aktivieren und Herunterfahren" gesetzt ist, wird dann, wenn der Dienst beim Erhalt des nächsten WOL-Pakets ausgeführt wird, ein Herunterfahren durchgeführt.
:end

:WOL_enable_help:
Wenn auf ja gesetzt, wird der Unraidwold-Daemon gestartet.
:end

:WOL_run_docker_help:
Wenn auf ja gesetzt, werden bei Empfang von Wake-on-LAN-Paketen Prüfungen für Docker durchgeführt. Andernfalls werden Docker ignoriert.
:end

:WOL_run_VM_help:
Wenn auf ja gesetzt, werden bei Empfang von Wake-on-LAN-Paketen Prüfungen für virtuelle Maschinen durchgeführt. Andernfalls werden virtuelle Maschinen ignoriert.
:end

:WOL_run_LXC_help:
Wenn auf ja gesetzt, werden bei Empfang von Wake-on-LAN-Paketen Prüfungen für LXC durchgeführt. Andernfalls wird LXC ignoriert. Das LXC-Plugin muss installiert sein, damit LXC verarbeitet wird.
:end

:WOL_run_shutdown_help:
Wenn auf ja gesetzt, werden bei Empfang von Wake-on-LAN-Paketen Prüfungen durchgeführt und wenn für den Dienst aktiviert und dieser ausgeführt wird, wird der Entität wird heruntergefahren. Das System muss auf "Aktiviert" und "Herunterfahren" gesetzt sein, damit es gehandelt wird.
:end

:WOL_interface_help:
Geben Sie die Schnittstelle an, die an den Daemon gebunden werden soll. Einige Schnittstellen können möglicherweise keine Etherframe-Pakete empfangen. Empfohlen ist, an eine physische Schnittstelle zu binden.
:end

:WOL_promiscuous_mode_help:
Aktivieren, um die NIC so einzustellen, dass sie Pakete nicht herausfiltert.
:end

:WOL_log_file_help:
Standardmäßig wird in die Syslog protokolliert. Falls Sie einen anderen Speicherort wünschen, legen Sie den Dateinamen fest.
:end
